# Redis 核心应用与分布式锁设计

## 一、Redis 核心数据结构与使用场景

### 1. String（字符串）
- **底层**：SDS（简单动态字符串）。
- **场景**：缓存对象（JSON序列化）、计数器（如文章点赞数 `INCR`）、分布式锁。

### 2. Hash（哈希）
- **特点**：类似Java的 `HashMap<String, Object>`，适合存储对象的部分属性。
- **场景**：存储用户信息，支持单独修改某个字段，无需整体序列化。

### 3. List（列表）
- **底层**：QuickList（压缩列表+双向链表）。
- **场景**：消息队列（`LPUSH` + `BRPOP`）、最新消息排行榜（如朋友圈时间线）。

### 4. Set（集合）
- **特点**：无序、元素唯一。
- **场景**：标签系统（交集、并集、差集）、抽奖（`SPOP` 随机抽取）、点赞去重。

### 5. ZSet（有序集合）
- **特点**：每个元素关联一个 `score`（分数），按分数排序。
- **底层**：跳表（Skip List）+ 哈希表。
- **场景**：排行榜（如游戏积分排名）、延时队列（将时间戳作为score）。

## 二、Redis 分布式锁设计（高并发下的库存扣减）

### 1. 为什么要分布式锁？
在单体应用中，我们可以用 `synchronized` 或 `ReentrantLock` 解决线程安全。但在多实例的微服务架构中，需要一种跨JVM的互斥机制来控制共享资源的访问，这就是分布式锁。

### 2. 基于Redis的分布式锁实现步骤（以Redisson为例）[citation:1]

#### 核心命令：`SET key value NX PX milliseconds`
- **NX**：只在键不存在时设置成功（互斥性）。
- **PX**：设置过期时间，防止死锁（即使客户端宕机，锁也会自动释放）。

#### 细节考量（三步走）
1.  **加锁的原子性**：必须使用 `SET NX PX` 一条命令完成，避免分两步执行（`SETNX` + `EXPIRE`）可能因服务器宕机导致死锁。
2.  **锁的标识（防误删）**：`value` 必须设置为一个唯一标识（如 `UUID:threadId`）。释放锁时，需要先通过 `get` 判断该标识是否属于自己，然后再删除。
    ```lua
    // 释放锁的Lua脚本（保证原子性）
    if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
    else
        return 0
    end
    ```
3.  **锁续期（Watch Dog）**：如果业务执行时间超过锁的过期时间，锁会自动释放，导致并发问题。Redisson 等客户端提供了一个“看门狗”线程，在锁未释放且业务未执行完时，定期（如每10秒）给锁续期（重置为30秒）。

### 3. 主从架构下的锁失效问题与RedLock
- **问题**：如果Redis是主从架构（Master-Slave），客户端A在Master上加锁成功，但Master还未将数据同步到Slave就宕机了。Slave升级为Master，此时客户端B也能加锁成功，导致互斥失效。
- **RedLock算法（红锁）**：在多个独立的Redis节点（通常为奇数个，如5个）上同时加锁。只有在**超过半数**（N/2+1）的节点上加锁成功，并且加锁总耗时小于锁的过期时间，才认为加锁成功。
  - *注意*：RedLock 较为复杂且存在争议，实际生产中如果业务允许偶发的锁失效，通常使用单节点或哨兵模式即可。

## 三、缓存常见问题解决方案

### 1. 缓存穿透
- **描述**：查询一个根本不存在的数据（如ID为-1的用户），请求直接打到数据库上。如果是恶意攻击，会导致数据库压力过大。
- **解决**：
  - **缓存空对象**：将 `null` 也缓存起来，但设置较短的过期时间。
  - **布隆过滤器**：在缓存之前加一层布隆过滤器，将所有可能存在的key哈希到一个足够大的bitmap中。不存在的请求会被拦截。

### 2. 缓存击穿
- **描述**：某一个**热点key**（如某明星绯闻）在缓存过期的瞬间，同时有大量请求打过来，全部穿透到数据库。
- **解决**：
  - **互斥锁**：当缓存失效时，只允许一个线程去查询数据库并重建缓存，其他线程等待（对应分布式锁的应用场景）。
  - **热点数据永不过期**：物理上不设置过期时间，但在value中逻辑上存储一个过期时间。通过异步线程去更新缓存。

### 3. 缓存雪崩
- **描述**：**大量key**同时过期，或者Redis宕机，导致海量请求直扑数据库。
- **解决**：
  - **过期时间打散**：给基础过期时间加上一个随机值，防止同一时间集体失效。
  - **搭建高可用集群**：Redis主从+哨兵或Redis Cluster。
  - **服务熔断/降级**：后端数据库无法支撑时，直接返回默认值或错误提示。
