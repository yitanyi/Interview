# 技术架构设计模式与演进

## 一、架构设计基本原则
### 1. SOLID 原则
- **单一职责**：模块仅有一个改变的原因
- **开闭原则**：对扩展开放，对修改关闭
- **里氏替换**：子类可替换父类
- **接口隔离**：不强迫实现无用方法
- **依赖倒置**：依赖抽象而非具体

### 2. 高内聚、低耦合
- **按业务边界拆分**：限界上下文（Bounded Context）
- **依赖方向控制**：上层依赖下层，避免循环依赖

### 3. 架构决策记录（ADR）
- **格式**：标题、状态、上下文、决策、后果
- **价值**：记录历史决策原因，便于新人理解

## 二、经典架构模式
### 1. 分层架构
- **表示层 → 业务逻辑层 → 数据访问层 → 数据库**
- **依赖倒置改进**：业务层定义接口，数据层实现

### 2. 六边形架构（端口与适配器）
- **核心域**：业务逻辑，不依赖任何外部
- **端口**：定义交互契约（接口）
- **适配器**：实现端口（如 HTTP 控制器、数据库仓储）

### 3. 事件驱动架构
- **核心组件**：事件生产者、事件总线、事件消费者
- **优点**：松耦合、易扩展、最终一致性
- **挑战**：事务一致性、消息顺序、重复消费

### 4. CQRS（命令查询职责分离）
- **命令模型**：处理写操作，事件溯源保存状态变更
- **查询模型**：专门优化的读模型（宽表、物化视图）
- **适用场景**：读写比差异巨大、复杂查询需求

## 三、架构演进策略
### 1. 单体架构 → 微服务
- **拆分时机**：团队规模扩大、独立部署需求、技术栈异构
- **拆分策略**：按业务能力、按子域（核心域、支撑域、通用域）
- **拆分陷阱**：分布式事务、网络延迟、运维复杂度

### 2. 绞杀者模式（Strangler Fig）
- **逐步替换**：新功能在新服务开发，通过网关路由逐步切换流量
- **数据迁移**：双写、回滚方案

### 3. 遗留系统现代化
- **代码重构**：提取模块、解耦依赖
- **API 封装**：为遗留系统提供 REST 接口
- **最终淘汰**：建立下线计划

## 四、架构评审与决策框架
### 1. 质量属性评估
- **性能**：吞吐量、延迟
- **可用性**：SLA、故障恢复时间
- **可扩展性**：水平扩展能力
- **安全性**：数据加密、访问控制
- **可维护性**：代码可读性、部署便捷性

### 2. 技术选型雷达
- **Adopt**：强烈推荐，已广泛验证
- **Trial**：值得尝试，在部分项目试点
- **Assess**：值得关注，待进一步评估
- **Hold**：谨慎使用，或已过时

### 3. 成本估算
- **开发成本**：人力、时间
- **运维成本**：服务器、带宽、第三方服务
- **机会成本**：延迟上线损失

## 五、云原生架构实践
### 1. 十二要素应用
- **基准代码**：一份代码多份部署
- **依赖**：显式声明依赖
- **配置**：环境变量存储配置
- **后端服务**：作为附加资源
- **构建、发布、运行**：严格分离
- **进程**：无状态
- **端口绑定**：通过端口暴露服务
- **并发**：通过进程模型扩展
- **易处理**：快速启动、优雅终止
- **开发/生产环境一致性**：尽量相同
- **日志**：作为事件流
- **管理进程**：一次性任务

### 2. 服务网格
- **数据面**：Envoy 代理
- **控制面**：Istio Pilot、Citadel
- **价值**：透明流量管理、安全通信、可观测性

### 3. Serverless 适用场景
- **事件处理**：文件上传处理、消息队列消费
- **API 后端**：低频调用、动态扩缩容
- **定时任务**：Cron Job
