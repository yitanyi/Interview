# 分布式系统一致性与共识算法深度剖析

## 一、分布式理论基础
### 1. CAP 定理
- **Consistency（一致性）**：所有节点同一时间数据一致
- **Availability（可用性）**：每个请求都能得到非错误响应
- **Partition Tolerance（分区容错）**：网络分区发生时系统仍能工作
- **CP 与 AP 的抉择**：ZooKeeper（CP） vs Eureka（AP） vs Nacos（可切换）

### 2. BASE 理论
- **Basically Available**：基本可用（允许降级）
- **Soft State**：软状态（中间状态可存在）
- **Eventually Consistent**：最终一致性

### 3. 一致性模型分级
- **强一致性**：写入后立即对所有读可见（如 Raft 线性一致性读）
- **弱一致性**：不保证立即可见
- **最终一致性**：经过一段时间后收敛一致
- **因果一致性**：有因果关系的操作保持顺序

## 二、共识算法详解
### 1. Paxos 算法
- **Basic Paxos**：Proposer、Acceptor、Learner 角色与两阶段提交（Prepare/Promise、Accept/Accepted）
- **Multi-Paxos**：优化领导选举，避免重复 Prepare
- **工程化难点**：活锁、日志空洞填补

### 2. Raft 算法（易于理解）
- **领导选举**：Term（任期）、随机超时触发选举、投票过半即当选
- **日志复制**：Leader 接收请求 → 追加日志 → 并发 AppendEntries RPC → 过半提交 → 状态机应用
- **安全性**：选举限制（Candidate 的日志必须至少和大多数节点一样新）
- **成员变更**：联合共识（Joint Consensus）避免双主

### 3. Zab 协议（ZooKeeper 专用）
- **崩溃恢复模式**：选举具有最大 ZXID 的节点为 Leader
- **消息广播模式**：两阶段提交变种（提议 → 过半 Ack → 提交）

## 三、一致性哈希与数据分片
### 1. 一致性哈希
- **原理**：将节点和 Key 映射到 0~2^32-1 的哈希环上
- **虚拟节点**：解决物理节点少时的数据倾斜问题
- **应用**：Redis Cluster 哈希槽（16384 个槽位）、Dubbo 负载均衡

### 2. 数据分片策略
- **Range 分片**：按主键范围划分（如 user_id 1~1000），易热点
- **Hash 分片**：均匀但范围查询困难
- **动态再平衡**：一致性哈希增量迁移 vs 哈希槽整体迁移

## 四、分布式事务方案选型
### 1. 两阶段提交（2PC）
- **协调者与参与者**：Prepare 阶段（写 undo/redo 日志）→ Commit/Rollback 阶段
- **缺陷**：同步阻塞、协调者单点故障、数据不一致风险

### 2. 三阶段提交（3PC）
- **CanCommit、PreCommit、DoCommit**：引入超时机制降低阻塞概率

### 3. TCC（Try-Confirm-Cancel）
- **Try**：预留资源（如冻结库存）
- **Confirm**：确认执行业务操作
- **Cancel**：回滚释放资源
- **实现难点**：空回滚、悬挂、幂等性保证

### 4. Saga 模式
- **正向补偿**：每个子事务都有对应的补偿操作
- **编排方式**：事件编排 vs 命令编排
- **适用场景**：长事务、跨服务调用链长

### 5. 本地消息表 + 消息队列
- **核心**：本地事务与消息持久化在同一数据库事务中，异步轮询发送消息保证最终一致性

## 五、分布式锁的深入实现
### 1. 基于 Redis 的 Redlock 算法争议
- **Martin Kleppmann 的质疑**：时钟跳跃导致锁失效
- **Redisson 红锁实现**：多数派加锁成功才返回
- **实际生产建议**：优先使用 ZooKeeper/Etcd 强一致锁

### 2. 基于 ZooKeeper 的分布式锁
- **临时顺序节点**：Watch 前一个节点，避免惊群效应
- **可重入性**：利用 ThreadLocal 记录锁次数

### 3. 基于 Etcd 的分布式锁
- **Lease 租约机制**：自动续期
- **事务支持**：原子化检查 Key 不存在则创建
