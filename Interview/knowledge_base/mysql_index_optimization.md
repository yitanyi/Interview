# MySQL索引优化与事务原理深度解析

## 一、索引底层数据结构：B+ Tree

### 为什么是B+ Tree？
- **磁盘IO优化**：B+Tree非叶子节点不存储数据，只存储索引（键值+指针），因此每个节点能存放更多的索引项。这意味着树的高度降低，从根节点到叶子节点的查找次数（IO次数）减少。
- **范围查询高效**：B+Tree的叶子节点之间通过指针连接成一个有序链表，非常适合范围查询（`BETWEEN`、`<`、`>`）。只需找到范围下限，然后顺序遍历叶子节点即可。
- **数据稳定性**：所有数据都存储在叶子节点，且叶子节点深度相同，查询任何一条记录所花费的IO次数基本相同。

### InnoDB索引实现（聚簇索引与二级索引）
- **聚簇索引**：InnoDB中，表数据文件本身就是按B+Tree组织的一个索引结构。**叶子节点保存了整行数据**。每个表必须有且仅有一个聚簇索引（通常是主键）。
- **二级索引**：叶子节点存储的是**主键值**。这意味着如果通过二级索引查找数据，先找到主键值，再通过聚簇索引回表查询完整数据。

## 二、索引设计原则与优化实战 [citation:1]

### 1. 什么时候该建索引？
- **频繁作为 `WHERE` 条件**的字段。
- **`ORDER BY` 或 `GROUP BY`** 的字段（利用索引的有序性避免文件排序）。
- **关联查询（JOIN）** 的关联字段。
- **区分度高的字段**（如身份证号 > 性别）。

### 2. 联合索引的最左前缀原则
- **定义**：如果创建了 `(a, b, c)` 的联合索引，那么实际上相当于创建了 `(a)`、`(a,b)`、`(a,b,c)` 三个索引。
- **匹配规则**：查询条件必须从索引的最左边列开始，并且不能跳过中间的列。
  - `WHERE a = 1 AND b = 2`：**有效利用**索引。
  - `WHERE b = 2 AND c = 3`：**无法利用**索引（因为跳过了a列）。
  - `WHERE a = 1 AND c = 3`：**只能利用到a列**，c列无法利用（因为跳过了b列）。

### 3. 索引失效的典型场景 [citation:1]
- **隐式类型转换**：例如字段是 `varchar` 类型，但传入的是数字 `123`。MySQL会将字段转换为数字进行比较，导致索引失效（相当于使用了 `CAST()` 函数）。
- **对索引列使用函数或计算**：`WHERE DATE(create_time) = '2026-02-24'` 应该改为 `WHERE create_time >= '2026-02-24 00:00:00' AND create_time < '2026-02-25 00:00:00'`。
- **`LIKE` 以通配符开头**：`WHERE name LIKE '%张三'` 失效；`WHERE name LIKE '张三%'` 有效（范围查询）。
- **`OR` 操作符**：`WHERE age = 18 OR name = '张三'`，如果 `age` 有索引但 `name` 没有索引，则两个条件都不会走索引。
- **联合索引违反最左前缀原则**（如上所述）。

## 三、事务隔离级别与MVCC

### 1. 事务的ACID特性
- **原子性**：通过 `undo log` 实现，记录修改前的数据，用于回滚。
- **一致性**：通过代码逻辑、约束以及其他三个特性共同保证。
- **隔离性**：通过 `锁` 和 `MVCC` 实现。
- **持久性**：通过 `redo log` 实现，记录修改后的数据，用于崩溃恢复。

### 2. 四种隔离级别及解决的问题
- **READ UNCOMMITTED**：脏读、不可重复读、幻读都存在。
- **READ COMMITTED**：解决了脏读，但不可重复读和幻读可能存在。
- **REPEATABLE READ（MySQL默认）**：解决了脏读和不可重复读，但幻读理论上存在（InnoDB通过间隙锁解决了幻读）。
- **SERIALIZABLE**：全部解决，但并发性能极低。

### 3. MVCC（多版本并发控制）
- **核心目的**：实现 **非阻塞读**。在 `REPEATABLE READ` 和 `READ COMMITTED` 级别下，普通的 `SELECT` 操作不加锁，通过读取数据行的快照版本来提高并发性能。
- **实现依赖于**：隐藏字段 `DB_TRX_ID`（最近修改事务ID）、`DB_ROLL_PTR`（回滚指针，指向 `undo log`）、`DB_ROW_ID`（隐藏主键）。
- **Read View**：
  - **READ COMMITTED**：每次执行 `SELECT` 都会生成一个新的 `Read View`，因此可以看到其他事务已提交的修改（导致不可重复读）。
  - **REPEATABLE READ**：**只在事务第一次执行 `SELECT` 时生成一个 `Read View`**，整个事务期间都使用这个视图，从而保证了可重复读。
